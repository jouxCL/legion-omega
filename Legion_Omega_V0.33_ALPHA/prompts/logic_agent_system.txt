### Clean Architecture (Feature-First)
- Strictly adhere to layers: Presentation, Domain, Data
- Dependencies always point inward (Presentation → Domain ← Data)
- Domain layer: entities, repository interfaces, use cases
- Data layer: repository implementations, data sources, models
- Presentation layer: UI, blocs/cubits, view models
- Each feature is a self-contained module with all three layers

### Directory Structure per Feature
lib/features/feature_name/

│   ├── models/            # DTOs with @JsonSerializable

    ├── pages/             # Screen widgets
    └── widgets/           # Feature-specific widgets

### flutter_bloc Implementation
- Use Cubit for simple state, Bloc for complex event-driven state
- Use Freezed for immutable states and union types
- Handle loading, error, and success states explicitly
- Avoid business logic in UI components
- Use BlocProvider for dependency injection of Blocs
- Separate event handling from UI logic

### Dependency Injection
- Use GetIt as service locator
- Register by feature in separate files
- Singletons for services and repositories
- Factories for transient objects (Blocs/Cubits)
- Lazy singletons for expensive services

### State Management
- States must be immutable using Freezed
- Use union types: initial | loading | success | error
- Emit specific typed error states with Failure details
- Use copyWith for state transitions
- Handle side effects with BlocListener
- Use BlocBuilder with buildWhen for optimized rebuilds

### Error Handling with Dartz
- Use Either<Failure, Success> for all repository and use case returns
- Left = Failure, Right = Success
- Create base Failure class, extend for specific types:
  - ServerFailure, CacheFailure, NetworkFailure, ValidationFailure
- Use fold() for handling both cases in one call
- Never throw exceptions across layer boundaries — map to Failure instead

```dart
abstract class Failure extends Equatable {
  final String message;
  const Failure(this.message);
  @override List<Object> get props => [message];
}
class ServerFailure extends Failure {
  const ServerFailure([String message = 'Server error occurred']) : super(message);
}
```

### Use Case Pattern
```dart
abstract class UseCase<Type, Params> {
  Future<Either<Failure, Type>> call(Params params);
}
// Example:
class GetUser implements UseCase<User, String> {
  final UserRepository repository;
  GetUser(this.repository);
  @override
  Future<Either<Failure, User>> call(String userId) async =>
      await repository.getUser(userId);
}
```

### Repository Pattern
```dart
abstract class UserRepository {
  Future<Either<Failure, User>> getUser(String id);
}
class UserRepositoryImpl implements UserRepository {
  final UserRemoteDataSource remoteDataSource;
  final NetworkInfo networkInfo;
  UserRepositoryImpl({required this.remoteDataSource, required this.networkInfo});
  @override
  Future<Either<Failure, User>> getUser(String id) async {
    if (await networkInfo.isConnected) {
      try {
        final remote = await remoteDataSource.getUser(id);
        return Right(remote.toDomain());
      } on ServerException {
        return Left(ServerFailure());
      }
    } else {
      return Left(NetworkFailure());
    }
  }
}
```

### Model and Database Conventions
- Include createdAt, updatedAt, isDeleted fields in all Firestore/API documents
- Use @JsonSerializable(fieldRename: FieldRename.snake) for all models
- Use @JsonKey(includeFromJson: true, includeToJson: false) for read-only fields
- Extend Equatable or use Freezed for value equality

### Enum Convention
- Use Enum instead of String where needed
- Write bool extension getters for every enum:
```dart

extension SomeEnumTypeX on SomeEnumType {
  bool get isType1 => this == SomeEnumType.type1;
  bool get isType2 => this == SomeEnumType.type2;
}
```
